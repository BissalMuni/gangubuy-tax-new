# 트리 구조 지식 관리 컨벤션

> **목적**: 지식을 트리 구조(책 → 카테고리 → … → 리프)로 관리하는 모든 프로젝트에 그대로 복사·적용 가능한 명명 규칙. 사이드바·페이지 헤딩 형식을 통일해 구조를 한눈에 파악할 수 있게 한다.
>
> **적용 범위**: 트리 데이터 파일(JSON/TS)의 노드 `title` 필드, 콘텐츠 파일(MDX/TSX) 내 헤딩.
>
> **버전**: 1.0 (2026-05-04)

---

## 1. 트리 구조 모델

### 용어
- **바구니 (basket)**: 책을 묶는 UX 라벨. 책 ID 참조만, URL/콘텐츠 없음. 한 책이 여러 바구니에 속할 수 있음.
- **책 (book)**: 학습 단위. URL·폴더·관리의 독립 기준. 트리의 루트 (depth 1).
- **노드 (node)**: 자식을 가지는 내부 요소 (책·중간 노드).
- **리프 (leaf)**: 자식이 없는 말단. 콘텐츠 1개 파일을 보유.

> 4계층 분리: **basket / book / content / map**.
> - basket → 책 묶음 (UX)
> - book → 트리 구조 (논리)
> - content → leaf 단일 TSX (AI 편집 단위)
> - map → book + leaf → content 라우팅 (컨벤션 기반)

### 최대 깊이: 5
트리는 최대 5단계까지. 자연 깊이가 더 얕으면 그 깊이에서 leaf로 둠 (인위 레이어 X).

### 깊이별 변형
| 깊이 | 노드 수 + 리프 | 구성 |
|---|---|---|
| 5 | 4 + 1 | 책 / 중단원 / 소단원 / 장 / 리프 |
| 4 | 3 + 1 | 책 / 중단원 / 장 / 리프 |
| 3 | 2 + 1 | 책 / 장 / 리프 |
| 2 | 1 + 1 | 책 / 리프 |

> 핵심 원칙: **장(章)은 리프 직상위 노드의 이름**. 트리 깊이가 어떻든 리프 바로 위는 항상 "장".

### 책 명명 권위 (Book Naming Authority)

새 책을 추가할 때 **정식 이름 한 개**를 정한다. 그 이름이 아래 모든 차원에 **동일하게 적용**된다:

| 차원 | 형식 |
|---|---|
| 콘텐츠 폴더 | `src/content/{이름}/` |
| 앱 라우트 폴더 | `src/app/{이름}/` |
| URL prefix | `/{이름}` |
| JSON `id` | `{이름}` |
| JSON `basePath` | `{이름}` |
| 데이터·로더 파일명 어간 | `{이름}` (권장) |

**표시명(`title`)만 분리** — UX용 자연어 이름은 자유롭게 결정.
예: id `"llm-learning"` ↔ title `"LLM 학습"` / id `"middle-school"` ↔ title `"중등수학"`

#### 왜 이 규칙이 필요한가
차원별로 다른 시점·다른 목표(URL은 짧게, 파일은 의미 있게)로 명명하면 시간이 지날수록 어긋난다. 한 이름으로 묶어두면 어느 한 차원만 봐도 나머지를 추측 가능. 리네임 비용·실수 모두 감소.

### 바구니 명명 권위 (Basket Naming Authority)

바구니도 책처럼 **정식 이름 한 개**를 정한다:

| 차원 | 형식 |
|---|---|
| 파일명 | `src/basket/{이름}.ts` |
| `id` 필드 | `{이름}` |

`title`은 UX용 자연어로 분리 (예: id `"ai"` ↔ title `"AI/LLM"`).

바구니는 책 ID를 **참조만**한다 (`bookIds: ["middle-school", ...]`). URL·콘텐츠 없음.
**모든 책은 최소 1개 바구니에 속함** (고아 책 금지 — 사이드바에 안 보임).

#### 바구니 뎁스 = 1 (평평)
바구니는 **다른 바구니를 담을 수 없음**. 항상 단일 레벨.
"중등수학을 수학 바구니의 하위 바구니로" 같은 중첩 구조는 도입하지 않는다.
세분화가 필요하면 **책의 트리 깊이**(최대 5)로 표현하거나, 책 자체를 더 작게 쪼갠다.

## 2. 노드 title 형식 (사이드바 표시)

### 깊이별 prefix
| 위치 | Prefix 형식 | 예 |
|---|---|---|
| 책 (depth 1) | 이름만 | "수학", "LLM" |
| 중단원 (depth 2, 5단계 트리) | prefix 없음 | "중등수학", "고등수학" |
| 소단원 (depth 3, 5단계 트리) | prefix 없음 | "중학교 1학년", "공통수학1" |
| **장** (리프 직상위) | `1장. 2장. 3장.` (Arabic + 장. + 이름) | "1장. 수와 연산", "2장. 문자와 식" |
| **리프** (말단) | `Ⅰ. Ⅱ. Ⅲ.` (Roman + 이름, 그룹 내 reset) | "Ⅰ. 소인수분해", "Ⅱ. 정수와 유리수" |

### 핵심 규칙
- **리프 = 로마자**: 같은 장 안에서 1부터 다시 시작 (글로벌 연속 번호 X).
- **장 = `N장.` Arabic**: 같은 부모 안에서 1부터 다시 시작.
- **그 외 상위 노드 = prefix 없음**: 자체 이름이 식별자 역할 (예: "공통수학1"은 이미 "1"을 포함).
- **slug는 prefix와 무관**: URL 경로용 slug는 변경하지 않음. `title`만 형식을 따름.

## 3. 콘텐츠 파일 헤딩 형식

각 리프는 콘텐츠 파일 1개를 가짐 (MDX/TSX 등). 파일 안의 헤딩 계층:

| 의미상 단계 | Prefix 형식 | 강제 여부 | 예 |
|---|---|---|---|
| 페이지 제목 (리프 title 자동 렌더) | `Ⅰ. xxx` (Roman, 사이드바와 동일) | 강제 | "Ⅰ. 소인수분해" |
| **h1** (콘텐츠 1단계 헤딩) | `1. xxx` (자연수 + 마침표) | **강제** (모든 1단계 헤딩) | "1. 소수와 합성수" |
| **h2** (콘텐츠 2단계 헤딩) | `(1) xxx` (괄호 숫자) | **강제** — SubSection에 title이 있으면 항상 prefix 부여 (헤딩이 필요 없으면 SubSection을 사용하지 않음). 부모 CalcBox 안에서 1부터 reset | "(1) 약수의 정의" |
| **h3** (콘텐츠 3단계 헤딩) | `① xxx` (원 숫자, 고정) | **고정 형식** — 3단계가 등장하면 반드시 `①②③` 사용 (다른 prefix 금지). 3단계 자체는 비강제 (필요 없으면 안 써도 됨) | "① 예시: 12의 약수" |

### 컴포넌트와 무관한 인라인 요소
다음 컴포넌트들은 헤딩 계층과 별개로 본문 내 어디서나 사용 가능 (번호 prefix 규칙 적용 안 됨):
- `Step` — 단계 번호 뱃지
- `Insight` — 핵심 인사이트 박스
- `Matrix` — 행렬/벡터 시각화
- `Arrow` — 화살표 연산자
- `BlockMath` / `InlineMath` — KaTeX 수식

### 주의
- "h1/h2/h3"는 **콘텐츠 내 1/2/3단계 헤딩**을 의미. 실제 HTML 태그는 페이지 제목이 h1이므로 한 단계씩 밀려 h2/h3/h4로 렌더링됨 (또는 그렇게 매핑되는 컴포넌트로).
- 같은 단계 안에서 번호는 1부터 다시 시작 (그룹 내 reset).
- 4단계 이상은 마크다운 리스트(`-`, `*`)나 본문 단락으로 처리. 별도 헤딩 컴포넌트 없음.

## 4. 새 프로젝트에 적용하는 절차

1. **책 이름 결정**: 정식 이름 한 개를 정한다 (§1 "책 명명 권위" 참조). 이 이름이 폴더·URL·id·basePath 모두에 동일 적용.
2. **트리 데이터 파일 작성**: 책 단위 JSON/TS 파일을 만들고 `id`, `title`, `slug`, `children` 필드로 트리 구성.
3. **깊이 결정**: 콘텐츠 양과 도메인 특성에 맞춰 트리 깊이(2~5) 결정. 5단계 한도.
4. **title 형식 적용**: 위 §2 표대로 prefix 부여. 리프는 Roman, 장은 `N장.`, 그 외는 이름만.
5. **콘텐츠 파일 작성**: 위 §3 헤딩 컴포넌트로 1/2/3단계 작성.
6. **컴포넌트 매핑**: 헤딩 컴포넌트(예: `CalcBox` → h2, `SubSection` → h3)와 prefix 형식을 프로젝트별 RULES에 명시.

## 5. 검증 체크리스트

새 책 추가 후 확인:

- [ ] 책 정식 이름이 폴더(`src/content/`, `src/app/`)·URL·JSON `id`·`basePath`·로더 파일 **모든 차원에 동일**한가?
- [ ] 표시명(`title`)만 자연어로 자유 결정되었는가?
- [ ] 책이 **최소 1개 바구니**(`src/basket/*.ts`의 `bookIds`)에 등록되었는가?
- [ ] `src/book/index.ts`의 `allBooks`에 추가되었는가?
- [ ] `src/map/index.ts`의 `derivePath` switch에 case 추가되었는가?

새 노드/콘텐츠 추가 후 확인:

- [ ] 리프 title이 같은 장 내에서 `Ⅰ.` 부터 1씩 증가하는가?
- [ ] 장 title이 같은 부모 내에서 `1장.` 부터 1씩 증가하는가?
- [ ] 책·중단원·소단원에 불필요한 prefix가 없는가?
- [ ] 콘텐츠 파일의 1단계 헤딩이 `1. xxx` 형식인가?
- [ ] 2단계 헤딩이 `(1) xxx` 형식인가?
- [ ] 3단계 헤딩이 `- xxx` 형식인가?
- [ ] slug는 변경하지 않았는가?

## 6. FAQ

**Q. 리프와 페이지 h1이 같은 형식(Roman)인 이유?**  
A. 리프 노드의 `title`이 사이드바와 페이지 제목 양쪽에 동일하게 렌더되기 때문. 둘을 다른 형식으로 두려면 추가 필드/변환 로직이 필요해서 단순함을 위해 동일 유지.

**Q. 왜 장은 Arabic인데 리프는 Roman인가?**  
A. 시각적 구분을 위해. 같은 사이드바에서 장 그룹과 리프를 한눈에 분리. 한국 교과서 관습(대단원 Ⅰ., 중단원 1.)과는 반대지만, 본 컨벤션에서는 가독성·구분 우선.

**Q. 장이 1개뿐이어도 `1장.` 붙이는가?**  
A. 붙인다. 장이 2개로 늘어날 미래를 대비하고 형식 일관성을 유지.

**Q. 깊이가 2(책 + 리프)인 경우 리프는 그대로 Roman?**  
A. 그렇다. 장 없이도 리프는 Roman 유지.
