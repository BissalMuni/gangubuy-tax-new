# 지방세 실무 정보 시스템 Constitution

## Core Principles

### I. 지방세 실무 정보 제공 (NON-NEGOTIABLE)

이 시스템의 목적은 **지방세 실무에 필요한 정확하고 구조화된 정보**를 제공하는 것이다.

- 제공 정보는 취득세, 재산세, 자동차세 등 지방세 전 영역을 포괄해야 한다
- 모든 정보는 현행 법령에 근거한 실무 적용 가능한 내용이어야 한다
- 이론 나열이 아닌 실무자가 즉시 참조할 수 있는 형태로 구성해야 한다
- 세율표, 감면 요건, 신고·납부 절차 등 핵심 실무 정보를 우선 구축한다

**근거**: 지방세 실무 담당자는 복잡한 세법을 빠르게 조회하고 적용해야 한다.
실무 중심 정보 구성이 시스템의 핵심 가치다.

### II. 대상 분리 (NON-NEGOTIABLE)

모든 콘텐츠는 반드시 **대상(audience)**이 명시되어야 한다.

- 콘텐츠 대상은 `internal` (세무부서 직원용) 또는 `public` (대시민공개용)으로 분류한다
- 현재 단계에서는 `internal` 콘텐츠만 생성한다
- `public` 콘텐츠는 반드시 `internal` 콘텐츠의 부분집합이어야 한다
- 대시민공개 전환 시 별도 검토 절차를 거쳐야 한다
- 대상 정보는 MDX frontmatter의 `audience` 필드에 명시해야 한다

**근거**: 직원용 상세 정보와 민원인용 안내 정보의 깊이와 범위가 다르다.
혼재 시 정보 노출 사고 또는 혼란이 발생할 수 있다.

### III. 정보 정확성 및 법적 준수 (NON-NEGOTIABLE)

모든 세금 정보는 법적 근거에 기반하며 법적 책임을 충족해야 한다.

- 세율, 과세표준, 감면 조건 등 수치 정보는 관련 법령을 출처로 명시해야 한다
  (예: "지방세법 제11조", "지방세특례제한법 제33조")
- 법령 개정 시 해당 콘텐츠를 즉시 갱신해야 한다
- 불확실한 해석이 포함된 경우 반드시 그 사실을 표기해야 한다
- 모든 공개 페이지에 법적 고지사항을 표시해야 한다:
  "이 시스템은 정보 제공 목적으로만 사용되며, 실제 세무 상담이나 법적 조언을 대체할 수 없습니다."
- 외부 기관(국세청, 행정안전부) 출처 데이터는 참조 링크와 함께 명시해야 한다

**근거**: 잘못된 세무 정보는 실무 오류와 민원을 초래한다.
정확성과 법적 준수는 시스템 신뢰의 기반이다.

### IV. MDX 기반 콘텐츠 관리 및 버전 관리

모든 세금 정보 콘텐츠는 **MDX 파일**로 관리하며 버전 이력을 유지한다.

- MDX 파일이 모든 세금 콘텐츠의 단일 출처(Single Source of Truth)다
- 모든 MDX 파일은 구조화된 frontmatter를 포함해야 한다 (Technical Standards 참조)
- 콘텐츠 업데이트 시 기존 파일을 수정하지 않고 새 버전 파일을 생성한다
  (예: `v1.0.0.mdx` → `v1.1.0.mdx`)
- 각 콘텐츠 디렉토리는 `_meta.json`으로 버전 이력을 관리해야 한다
- 모든 버전은 `effectiveDate` (법적 시행일)를 명시해야 한다

**근거**: 세법은 자주 개정되며 특정 시행일이 있다. 불변 버전 관리는
역사적 정확성과 법적 감사 추적을 보장한다.

### V. 개조식 공문서 스타일

모든 세무 콘텐츠는 **개조식 공문서 형식**을 따라야 한다.

- 서술형 문장 대신 번호/글머리 기호를 사용한 구조화된 형식을 사용한다
- 세율, 요건 등은 반드시 표(table) 형식으로 정리한다
- 법령 근거를 반드시 명시한다
- 전문적이고 간결한 행정 용어를 사용한다

**근거**: 세무 문서는 정확성과 일관성이 핵심이며,
개조식은 정보 검색과 비교에 최적화된 형식이다.

### VI. 사용자 피드백 기반 콘텐츠 업데이트

콘텐츠는 **사용자 피드백**을 통해 지속적으로 개선된다.

- 사용자(세무부서 직원)가 오류, 누락, 개선 사항을 제보할 수 있는 경로를 제공해야 한다
- 제보된 피드백은 검토 후 새 버전 파일로 반영한다 (Principle IV 준수)
- 피드백 반영 시 변경 내용과 근거를 `_meta.json`의 changelog에 기록해야 한다
- 빈번히 조회되거나 피드백이 집중되는 항목을 우선 고도화한다

**근거**: 실무 담당자의 직접 피드백이 콘텐츠 품질을 높이는 가장 효과적인 방법이다.
사용자 피드백 루프를 시스템에 내재화하여 지속 개선을 보장한다.

### VII. 단순성

구현은 현재 필요한 **최소한의 복잡도**만 허용한다.

- 가상의 미래 요구사항을 위한 추상화를 만들지 않는다 (YAGNI)
- 한 번만 쓰이는 로직을 위한 유틸리티/헬퍼를 만들지 않는다
- 외부 의존성 추가 시 반드시 명확한 이유가 있어야 한다
- 새로운 추상화 계층이나 서드파티 서비스 도입 시 근거를 문서화해야 한다

**근거**: 일인개발 프로젝트에서 과도한 엔지니어링은 유지보수 부담을 증가시킨다.

### VIII. 점진적 공개

콘텐츠와 기능은 **단계적**으로 공개한다.

- **1단계**: 세무부서 직원용(`internal`) 콘텐츠 완성
- **2단계**: 직원용 콘텐츠 중 대시민공개 적합 항목 선별 및 검토
- **3단계**: 선별된 콘텐츠를 `public`으로 전환하여 공개
- 각 단계는 이전 단계가 완료된 후 진행해야 한다

**근거**: 직원용 정보 체계를 먼저 확립한 후 공개 범위를 결정하는 것이
정보 품질과 보안 측면에서 안전하다.

### IX. 개발 자동화 — Docker Claude Dangerous Mode

**Claude Code dangerous mode의 Docker 컨테이너 격리 자동화**를 개발 워크플로우로 채택한다.

- Claude Code를 `--dangerously-skip-permissions` 모드로 실행할 때 반드시
  Docker 컨테이너 내에서 격리 실행해야 한다
- `Dockerfile.claude` + `docker-compose.claude.yml` 기반 격리 환경을 표준으로 한다
- 컨테이너는 최소 권한 원칙을 적용해야 한다 (`cap_drop: ALL`, 필요 capability만 추가)
- 자동화 실행 결과는 `.claude-output/result.md`와 `follow-up.md`로 저장해야 한다
- `Dockerfile.claude`, `docker-compose.claude.yml`, `.claude-output/`은 `.gitignore`에
  포함하여 저장소에 커밋하지 않는다

**근거**: dangerous mode는 파일 시스템에 무제한 접근 권한을 부여한다.
Docker 격리를 통해 호스트 시스템 보호와 재현 가능한 자동화 환경을 동시에 확보한다.

### X. 권한 분리 (Permission Separation) (NON-NEGOTIABLE)

설계자(체계) / 담당자(내용) / 승인자(검토) / AI(보조)의 4개 역할을 분리한다.

- 단일 사용자가 둘 이상의 역할을 동시에 보유할 수 있다 (예: 관리자 + 승인자)
- 그러나 한 *행위*(상태 전이·매니페스트 변경·승인)는 하나의 *행동 컨텍스트*(acting_role)로만 수행되며, 감사 로그에 기록한다
- **자기-승인 금지** (Separation of Duties): 작성자(actor)와 승인자(reviewer)는 동일인일 수 없다. 1인 운영 비상 우회는 명시적 emergency 토글로만 가능하며 감사 로그에 강조 표시
- 구체 스펙: [specs/003-role-based-approval/spec.md](../../specs/003-role-based-approval/spec.md) FR-001a, FR-001c, FR-010a

**근거**: 정책 [docs/policy/manual-system-plan.md](../../docs/policy/manual-system-plan.md) §II.1·§IV의 "체계는 설계자, 내용은 담당자, AI는 보조" 원칙을 코드로 강제. 단일 사용자가 모든 모자를 쓰면 검토 누락이 발생하므로 보유와 행위를 분리하고 자기-승인을 막는 것이 신뢰의 기반.

### XI. 체계와 내용의 분리 (Structure-Content Separation) (NON-NEGOTIABLE)

메뉴 트리·파일 경계·식별자(체계)는 설계자가, MDX 본문(내용)은 담당자가 관리한다. 두 영역은 단일 매니페스트 JSON으로만 연결된다.

- 매니페스트 (`config/tree-manifest.json`)가 트리·콘텐츠 연결의 단일 진실의 출처
- 핵심 불변식: **1 트리 리프 = 1 contentId = 1 MDX 파일**. CI에서 강제
- ID는 `kebab-slug` 전역 유니크, 발급 후 영구 불변, 재사용 영구 금지 (retire 시 tombstone)
- order_label·label·content_path는 `mode:refactor` PR로만 일괄 변경 가능. 일반 PR에서는 변경 차단
- 구체 스펙: [specs/004-content-structure-separation/spec.md](../../specs/004-content-structure-separation/spec.md)

**근거**: 정책 §III.1의 "콘텐츠 구조 설계와 내용 채움의 분리". 이전 구조(`tree.ts` path 문자열)는 한쪽이 깨져도 빌드 통과하여 고아·죽은 링크가 누적됐다. 매니페스트 + CI 검증이 두 역할의 독립 작업을 안전하게 보장한다.

## Governance

### 원칙 수정 절차

1. 수정 제안과 근거를 문서화
2. 이 파일을 먼저 수정하고 관련 산출물에 반영
3. 버전 번호 증가 (아래 규칙 적용)
4. 관련 템플릿 및 커맨드 파일 동기화 확인

### 버전 증가 규칙

- **MAJOR**: 원칙 제거 또는 하위 호환 불가능한 변경
- **MINOR**: 원칙 추가 또는 실질적 내용 확장
- **PATCH**: 명확화, 오타 수정, 비의미론적 개선

### 준수 확인

- **모든 PR**: 파일 구조, 네이밍 컨벤션, frontmatter 스키마 자동 검사
- **원칙 II (대상 분리)** 위반은 어떤 경우에도 허용하지 않는다
- **원칙 III (정보 정확성)**: 세율/법령 변경 콘텐츠는 법적 근거 명시 필수
- **원칙 X (권한 분리)**: 자기-승인은 emergency 토글 외에 거부. acting_role 기록 누락 PR 머지 차단
- **원칙 XI (체계/내용 분리)**: 매니페스트 ↔ MDX 1:1 매칭, ID 유일성·불변성 검증 실패 PR 머지 차단
